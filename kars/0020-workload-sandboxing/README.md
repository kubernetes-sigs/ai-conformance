# KAR-0020: Workload Sandboxing for Untrusted Code

## Description

Provide a mechanism to run workloads in a sandbox that provides strong isolation from the host node kernel. This is crucial for AI agent use cases that may execute untrusted code. The sandbox should isolate the workload's process, memory, filesystem, and network access from the host node and other co-located pods.

> Scope: This requirement addresses directional containment—protecting the host infrastructure and co-located tenant workloads from a potentially malicious sandboxed workload. It does not provide confidentiality or integrity guarantees for the workload from the host node operator or cloud provider. Protecting workload data-in-use from the host infrastructure is the domain of confidential computing (TEEs + attestation) and is explicitly out of scope for this requirement.

## Motivation

AI agentic workloads often execute arbitrary, untrusted code generated by LLMs or fetched from external sources. Without strong isolation, compromised workloads can access host node resources, inspect memory of other pods, modify host filesystems, or initiate unauthorized network connections. Implementing robust workload sandboxing (such as microVM-based or user-space sandboxing via `RuntimeClass`) provides a secure boundary, safeguarding the host infrastructure and ensuring multi-tenant isolation.

The Kubernetes community has actively invested in tooling and runtimes for these patterns (e.g., Kata Containers, gVisor, and higher-level orchestration controllers like [agent-sandbox](https://github.com/kubernetes-sigs/agent-sandbox)). Demonstrating these community patterns highlights the viability and importance of strong workload isolation within the Kubernetes ecosystem.

## Test Plan

### Conformance Verification Scope

*Note: As established in community consensus, AI conformance strictly verifies the underlying platform capability to provide abstract kernel isolation mechanisms (e.g., via `RuntimeClass`), rather than mandating any specific higher-level controller (such as `agent-sandbox`) or runtime implementation.*

*Note on GPU and Hardware Isolation: While `RuntimeClass` provides an abstract mechanism for workload isolation, different container runtimes employ distinct architectural models for handling hardware accelerators and PCIe devices (e.g., userspace ioctl filtering via gVisor's `nvproxy` versus hardware-enforced IOMMU/VFIO passthrough via Kata Containers). Cluster operators evaluating workload sandboxing for GPU-enabled AI workloads should carefully assess their infrastructure threat model (such as bare-metal multi-tenancy versus nested virtualization) to select the runtime implementation and hardware isolation topology that aligns with their security requirements.*

### How We Might Test It

Validate the following observable outcomes by configuring a sandboxed `RuntimeClass` (e.g., `kata`, `gvisor`, or a platform-specific equivalent) and deploying a verification pod:

1. A pod specifying the sandboxed `RuntimeClass` is successfully scheduled and executed within the designated isolation boundary.
1. Probing scripts executed within the sandbox cannot access host process namespaces, inspect host memory tables, or mount host filesystem paths, confirming strict boundary separation from the host kernel and other tenant workloads.
1. The sandboxed workload is restricted to its designated network namespace, preventing unauthorized access to host-level network interfaces or host loopback traffic.

### Automated Tests

An automated conformance test could verify the observable outcomes above:
1. Leverage standard `RuntimeClass` definitions (or a configurable sandboxed runtime handler) to dynamically provision a sandboxed test pod.
1. Execute verification scripts within the test pod to confirm process, memory, filesystem, and network isolation from the host node (e.g., verifying distinct kernel versions, namespace separation, blocked host mounts, and restricted egress).
1. If a platform employs a specialized or unrecognized sandboxing mechanism that cannot be automatically verified, the test suite will output an "Unknown" status, signaling that manual attestation is required.
