# KAR-0020: Workload Sandboxing for Untrusted Code

## Description

Provide a mechanism to run workloads in a sandbox that provides strong isolation from the host node kernel. This is crucial for AI agent use cases that may execute untrusted code. The sandbox should isolate the workload's process, memory, and network access from the host node.

## Motivation

AI agentic workloads often execute arbitrary, untrusted code generated by LLMs or fetched from external sources. Without strong isolation, compromised workloads can access host node resources, inspect memory of other pods, or initiate unauthorized network connections. Implementing robust workload sandboxing (such as VM-based or user-space sandboxing) provides a secure boundary, safeguarding the host infrastructure and ensuring multi-tenant isolation.

The Kubernetes community has actively invested in tooling for these patterns through the [agent-sandbox](https://github.com/kubernetes-sigs/agent-sandbox) project. By providing a common CRD and controller architecture for sandboxed agent execution, agent-sandbox (along with Kata and gVisor runtimes) demonstrates the viability and importance of strong workload isolation within the Kubernetes ecosystem.

## Test Plan

### How We Might Test It

Validate the following observable outcomes by deploying [agent-sandbox](https://github.com/kubernetes-sigs/agent-sandbox) or configuring a sandboxed RuntimeClass (e.g., kata or gvisor):

1. A pod specifying the sandboxed RuntimeClass is successfully scheduled and executed within the designated isolation boundary.
1. Probing scripts executed within the sandbox cannot access host process namespaces or inspect host memory tables, confirming strict boundary separation from the host kernel and other tenant workloads.
1. The sandboxed workload is restricted to its designated network namespace, preventing unauthorized access to host-level network interfaces or host loopback traffic.

### Automated Tests

An automated conformance test could verify the observable outcomes above:
1. Leverage [agent-sandbox](https://github.com/kubernetes-sigs/agent-sandbox) or standard RuntimeClass definitions to dynamically provision a sandboxed test pod.
1. Execute verification scripts within the test pod to confirm process, memory, and network isolation from the host node (e.g., verifying distinct kernel versions, namespace separation, and blocked host access).
1. If a platform employs a specialized or unrecognized sandboxing mechanism that cannot be automatically verified, the test suite will output an "Unknown" status, signaling that manual attestation is required.
